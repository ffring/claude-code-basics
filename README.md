# Claude Code: методология работы с агентом

Полная методичка по работе с Claude Code как с инженерным агентом: от установки до автоматизированного пайплайна. Восемь последовательных шагов, каждый строится на предыдущем. На странице — теория из презентаций и практические упражнения по каждому занятию.

Сквозной кейс — автоматический AI-дайджест: блог, который сам ищет новости, пишет обзоры, генерирует обложки, коммитит в Git и публикует выпуски.

## Принцип

Работа с агентом — не «попросил — получил». Это конструирование среды, в которой агент действует точно: правильный контекст, точные инструменты, понятные правила, автоматические проверки. Эта среда называется **Agent Harness** — обвязка вокруг модели. Методология собирает её слой за слоем.

| Шаг | Слой | Что добавляется в harness |
|-----|------|---------------------------|
| 1 | Окружение | Установка, права доступа, базовый `CLAUDE.md` |
| 2 | Промптинг | Формат задачи: цель, контекст, ограничения, проверка |
| 3 | Планирование | Plan Mode — сначала план, потом правки |
| 4 | Контекст | Память, правила, постепенное раскрытие знаний |
| 5 | MCP | Подключение внешних сервисов и источников данных |
| 6 | Навыки и хуки | Повторяющиеся действия и автоматические проверки |
| 7 | Оркестрация | Несколько агентов параллельно, изоляция через worktrees |
| 8 | Harness | Полный цикл, отладка, headless, SDK, CI/CD |

Папки `01-base`...`08-harness` содержат оригинальные презентации в PDF и упражнения. Этот документ — текстовая выжимка по тем же материалам.

## Сквозной проект: AI-дайджест

Проект растёт по шагам:

1. Репозиторий и базовые редакционные правила.
2. Промпты для поиска, отбора и подготовки статей.
3. Архитектура сайта и порядок публикации.
4. Правила и знания вынесены из контекста в память и отдельные файлы.
5. Подключены Tavily (поиск), Replicate (графика), GitHub (репозиторий).
6. Повторяющиеся действия оформлены как навыки и хуки.
7. Работа разделена между агентами: поиск, текст, обложки, страницы.
8. Весь цикл запускается автоматически.

## Требования

- macOS, Windows или Linux. VS Code или другая IDE.
- Git и базовые навыки терминала.
- Доступ к Claude Code.
- Для практики — API-ключи Tavily, Replicate, GitHub, Vercel.

## Занятие 1. Основы

### Теория

**Что такое Claude Code.** Это агентский слой для работы с моделями Anthropic. Не сама модель, а агент в терминале, который выбирает модель, управляет инструментами и работает на вашем компьютере. Рабочая среда состоит из трёх компонентов: VS Code (редактор и терминал), Claude Code (агент) и AI-модель, которая обрабатывает запросы.

**Установка.**
- macOS (Homebrew): `brew install --cask claude-code`
- macOS / Linux / WSL: `curl -fsSL https://claude.ai/install.sh | bash`
- Windows CMD: `curl -fsSL https://claude.ai/install.cmd -o install.cmd && install.cmd && del install.cmd`

После установки запускается командой `claude`. Проверка установки — `claude doctor` (проверяет инсталляцию, аутентификацию, подключение к модели, разрешения, версию Node.js — нужен 18+).

**Подключение модели.** Подписка Claude Pro/Max — авторизация через браузер при первом запуске. Альтернативные провайдеры (Z.AI / GLM, MiniMax, OpenRouter) подключаются через переменные окружения:

```bash
export ANTHROPIC_BASE_URL=https://api.z.ai/api/anthropic
export ANTHROPIC_AUTH_TOKEN=ваш-ключ
```

Чтобы переменные сохранялись между сессиями — добавить в `~/.zshrc` (macOS/Linux) или через `[System.Environment]::SetEnvironmentVariable(...)` в PowerShell (Windows).

**Модели.**

| Модель | Когда использовать |
|---|---|
| Opus | Архитектура, рефакторинг, многофайловые изменения, сложный анализ |
| Sonnet | Повседневная работа, по умолчанию (80% задач) |
| Haiku | Простые вопросы, быстрые проверки |

Переключение внутри сессии — `/model`.

**Папка `.claude/`** — центр управления проектом:
- `settings.json` — настройки проекта: разрешения, хуки, переменные окружения. Попадает в Git.
- `settings.local.json` — личные настройки. Не попадает в Git (auto-gitignore).
- `rules/` — правила по темам, отдельные `.md`-файлы (`testing.md`, `api-design.md`).
- `skills/`, `agents/`, `worktrees/` — навыки, субагенты, рабочие копии (заполняются на следующих ступенях).

**CLAUDE.md.** Markdown-файл, который Claude читает при каждом запуске сессии. Самая высокорычажная точка настройки: плохая строка в коде ломает одну строку, плохая строка в CLAUDE.md влияет на каждый запрос. Целевой размер — около 200 строк, начинать с 5–10 правил.

**4 уровня иерархии CLAUDE.md** (по убыванию приоритета):

| Уровень | Путь | Git | Назначение |
|---|---|---|---|
| Managed | `/Library/.../ClaudeCode/CLAUDE.md` | — | Корпоративные политики, нельзя отключить |
| Глобальный | `~/.claude/CLAUDE.md` | — | Личные предпочтения для всех проектов |
| Проектный | `./CLAUDE.md` | Да | Контекст проекта: стек, структура, правила |
| Локальный | `./CLAUDE.local.md` | Нет | Персональные заметки по проекту |

Claude обходит дерево каталогов от текущей папки вверх и подгружает CLAUDE.md из каждой директории. В монорепо — отдельный CLAUDE.md в каждом пакете. При конфликте побеждают более специфичные инструкции.

**Структура WHY / WHAT / HOW.**
- **WHY** — зачем проект существует: цели, пользователи, бизнес-контекст.
- **WHAT** — что внутри: стек, структура папок, стиль кода, зависимости.
- **HOW** — как работать: команды сборки, процесс ветвления, ограничения, gotchas.

**Что включать:** команды сборки и запуска, рабочий процесс по шагам, доменную терминологию, типичные ошибки. **Что НЕ включать:** правила форматирования (для этого есть линтеры), секреты и API-ключи, общие советы вроде «пиши качественный код», описание файлов, которое видно из `package.json`.

Тест для каждой строки: «Если я уберу это, Claude будет делать ошибки?» Если нет — удалить.

**Инструменты CLAUDE.md.** `/init` — автогенерация стартового CLAUDE.md по проекту (всегда черновик, сокращайте). `#` — быстрая заметка прямо из сессии. `@path/to/file` — импорт файлов в CLAUDE.md, до 5 уровней вложенности. `.claude/rules/*.md` — модульные правила, загружаются при старте.

**Permission Modes.** Пять режимов доступа:

| Режим | Что делает | Когда использовать |
|---|---|---|
| Default | Спрашивает разрешение на каждое действие | Старт работы, чувствительные проекты |
| Accept Edits | Правки файлов без вопросов, Bash спрашивает | Повседневная работа |
| Plan | Только анализ и чтение, ничего не меняет | Code review, исследование кодовой базы |
| Don't Ask | Автоматически отклоняет всё, что не разрешено явно | Строгий режим |
| Bypass | Полный автопилот, флаг `--dangerously-skip-permissions` | Скрипты, CI |

Shift+Tab переключает первые три по кругу: Default → Accept Edits → Plan → Default. Текущий режим показан внизу терминала.

**Allow / Ask / Deny.**
- **Allow** — без подтверждения.
- **Ask** — спрашивать каждый раз.
- **Deny** — полная блокировка.

Порядок оценки: `deny → ask → allow`. Deny всегда побеждает. Правила из разных settings-файлов мёрджатся, проектные побеждают глобальные.

**Паттерны.** Пробел перед `*` важен: `Bash(npm run *)` совпадёт с `npm run dev`, `Bash(ls *)` совпадёт с `ls -la` но не с `lsof`. `*` — файлы в одной директории, `**` — рекурсивно. Паттерны работают для разных инструментов: `Edit(/docs/**)`, `Read(./.env)`, `WebFetch(domain:example.com)`, `mcp__puppeteer`, `Agent(Explore)`. Безопасность: `Bash(safe-cmd *)` НЕ совпадает с `safe-cmd && malicious-cmd`.

`/permissions` — показать все правила и из какого settings.json они пришли.

**Базовые команды.**

| Команда | Что делает |
|---|---|
| `/clear` | Полный сброс контекста между несвязанными задачами |
| `/compact` | Сжатие контекста внутри задачи, можно с фокусом: `/compact работаем над авторизацией` |
| `/cost` | Расход токенов и стоимость за сессию |
| `/model` | Переключить модель (Opus/Sonnet/Haiku) |
| `/permissions` | Просмотр правил доступа |
| `/memory` | Постоянная память |
| `/btw` | Быстрый вопрос вне контекста, не попадает в историю |
| `/rename` | Дать сессии понятное имя |
| `/help` | Список всех команд |
| `/init` | Сгенерировать стартовый CLAUDE.md |
| `/config` | Интерактивная настройка |

**Горячие клавиши.**
- `!команда` — прямой доступ к терминалу без участия Claude (не тратит токены).
- `Escape` — прервать выполнение. Двойной Escape — меню `/rewind`.
- `Ctrl+V` — вставить текст или изображение (именно Ctrl, даже на macOS; Cmd+V не работает для картинок).
- `#` — заметка в CLAUDE.md прямо из сессии.

**Управление сессиями.**
- `claude --continue` — продолжить последнюю сессию.
- `claude --resume` — выбрать из списка предыдущих сессий.
- `/rename "имя"` — дать сессии понятное имя для поиска.

Сессии хранятся локально, каждая — отдельный контекст, как ветка в Git. Также можно подавать контекст через путь к файлу, перетаскивание в терминал, `cat error.log | claude -p "объясни ошибки"`.

**Типичные ошибки новичков:** не используют `/clear` между задачами (контекст засоряется, появляется «призрачный контекст»); не смотрят `/cost`; не жмут Escape, когда Claude пошёл не туда; не дают контекст в промпте.

### Упражнение 1.1: Установка и проверка Claude Code

**Цель:** работающая среда — VS Code + Claude Code + подключённая модель.

```bash
/help
ls -la .claude/
```

**Критерий выполнения:** `claude doctor` проходит все 5 проверок без ошибок. Claude отвечает на промпты и видит содержимое рабочей папки.

**Цель:** настроить разрешения так, чтобы безопасное выполнялось без вопросов, опасное блокировалось.

## Занятие 2. Промптинг

### Теория

**Чем промпт для агента отличается от промпта для чатбота.** В чате расплывчатый запрос даёт длинный размытый ответ — вы перечитываете и уточняете. С кодовым агентом расплывчатость превращается в неправильные предположения, вшитые в кодовую базу: Claude меняет файлы, запускает команды, и цена ошибки — откат изменений. Поэтому для агента нужна другая дисциплина промпта.

**Структура агентного промпта — 5 элементов.** Любой рабочий запрос собирается из пяти блоков:

1. **Контекст** — зачем и почему. Не «что сделать», а «какую проблему решаем». Описывайте симптом и причину, а не только действие. Золотое правило: пишите, как для нового умного коллеги, который не знает ваш проект.
2. **Ссылки на код** — паттерн, на который опираться. Через путь («Следуй паттерну из `/api/apply/route.ts`»), через `@`-упоминание (`@components/ServiceCard.tsx`), по концепции («Повтори обработку ошибок из наших API-роутов»).
3. **Задача** — одно действие, не пять. Конкретный результат.
4. **Формат** — JSON, markdown, компонент, инлайн-ошибка под полем.
5. **Ограничения** — чего НЕ делать. «Без новых пакетов», «не трогай базу», «работай в существующих файлах».

**Три уровня конкретности.** Выбор уровня зависит от того, насколько вы готовы к свободе Claude.

| Уровень | Пример | Когда |
|---|---|---|
| Исследование | «Разберись, как устроена авторизация» | Хотите найти то, о чём не подумали |
| Направление | «Добавь кэширование, но не меняй API» | Знаете направление, не знаете деталей |
| Спецификация | «Замени fetch на axios, сохрани типы ответов» | Точно знаете, что нужно |

Правило: если вас будет раздражать неправильный выбор — переходите на уровень спецификации.

**Constraints-First: ограничения важнее инструкций.** Claude склонен к переусложнению — добавляет спиннеры, тосты, обработчики ошибок, утилиты, новые файлы. Без ограничений простой компонент превращается в 3–5 файлов. С ограничениями («не создавай новых файлов», «не добавляй пакетов», «не трогай базу») Claude делает ровно то, что попросили, с первой попытки. Описать, чего НЕ делать, часто важнее, чем описать, что нужно.

**Verification-Driven Prompts.** Anthropic называет это «самой эффективной техникой». Идея: дать Claude способ себя проверить — тест-кейсы, `npm run build`, скриншот, curl для API. Без верификации Claude создаёт правдоподобное, но непроверенное решение. С верификацией он запускает тесты, видит результат и сам исправляет ошибки до того, как покажет вам.

Пример блока верификации:
```
# Верификация
- user@example.com → true
- "invalid" → false
- user@.com → false
- "" → false
Запусти тесты после реализации.
```

**Управление глубиной рассуждения.** Extended Thinking включён по умолчанию. Управляющие механизмы:
- `Option+T` (macOS) / `Alt+T` — переключить thinking.
- `/effort` с уровнями `low`, `medium`, `high`, `max` — основной механизм.
- `MAX_THINKING_TOKENS=10000` — потолок бюджета.
- `Ctrl+O` — открыть транскрипт сессии с размышлениями.
- Ключевое слово `ultrathink` в промпте — включает high effort на следующий ход. «think», «think hard» — обычная инструкция, не выделяет thinking-токены.

**Planning vs. Thinking.** Это разные оси: Plan Mode (`Shift+Tab`) — про **широту** (breadth), многошаговые задачи и понимание незнакомой кодовой базы. Thinking — про **глубину** (depth), сложную логику, отладку. На сложной задаче в незнакомом коде включают оба.

**Git как страховка.** Claude меняет файлы напрямую — работа без Git это работа без права на ошибку. Claude управляет Git через естественный язык: «создай ветку», «закоммить», «открой PR», «покажи diff». При этом он никогда не коммитит без подтверждения.

Базовый workflow:
- **Именование веток**: `тип/короткое-описание` (`feature/user-auth`, `fix/login-redirect`, `hotfix/null-pointer`). Тип Claude подбирает сам из описания.
- **Атомарные коммиты**: одно изменение = один коммит. «Разбей изменения на логические коммиты: API, UI-компоненты, тесты».
- **Conventional Commits**: `тип(область): описание`, до 72 символов в subject. Типы: `feat`, `fix`, `docs`, `refactor`, `test`, `chore`.
- **`/commit`** — Claude читает diff, предлагает сообщение, ждёт подтверждения.
- **`/commit-push-pr`** — полный цикл: ветка + коммит + push + PR за одну команду.
- **Pre-Push Code Review**: «Review changes on this branch vs main. Ищи: баги, безопасность, обработку ошибок» — перед пушем.
- **Откат**: двойной `Escape` → `/rewind` → восстановить код, разговор или оба. `git checkout` — не для ручной починки, проще откатить, чем чинить. Для возврата к закрытой сессии — `/resume`.

Git-конвенции записываются один раз в `## Git Conventions` секцию `CLAUDE.md` — Claude следует им автоматически у всей команды.

**8 анти-паттернов промптинга.**

| Анти-паттерн | Симптом | Решение |
|---|---|---|
| Мега-промпт | 5 фич в одном сообщении, качество падает по всем | 1 фича = 1 промпт + коммит между |
| Kitchen Sink | Несвязанные задачи в одной сессии, контекст забит мусором | `/clear` между темами |
| Призрачный контекст | «Claude забыл конвенцию» | Постоянное — в `CLAUDE.md` |
| Fix Loop | Исправления исправлений ломают что-то новое | `Escape` + `/clear` + новый промпт |
| Iterative Refinement | 5+ правок не улучшают результат | `/clear` + лучший начальный промпт |
| Infinite Exploration | Claude читает сотни файлов, контекст забит | Ограничить область поиска |
| Over-Specified CLAUDE.md | Claude игнорирует часть правил | <100 строк, тест «уберу — будет ошибаться?» |
| Trust-Then-Verify Gap | Код не работает в граничных случаях | Всегда давать способ проверки |

**Правило двух попыток.** После двух неудачных попыток — `/clear` и новый промпт. Чистая сессия с уточнённым промптом почти всегда лучше длинной сессии с накопленными правками. Вкладывайте время в начальный промпт, а не в исправления.

**Сигналы проблемы.** Claude повторяет одну ошибку → Fix Loop. Ответы стали длиннее и менее точные → контекст перегружен (`/compact` или `/clear`). Claude добавляет незапрошенные фичи → нет ограничений. `/cost` показывает аномальный расход → пора `/clear`.

**`Escape` vs. `/compact` vs. `/clear`.**
- `Escape` — прервать выполнение, контекст сохранён.
- `/compact` — сжать контекст, сохранив суть. Когда задача та же, но сессия растёт.
- `/clear` — полный сброс. Новая задача или 2+ неудачных попытки.

## Занятие 3. Планирование

### Теория

**Agentic Loop: как Claude Code работает под капотом**

Claude Code — не сама модель, а обвязка (harness), которая управляет моделью через вызовы инструментов. Модель умеет только генерировать текст. Чтение файлов, запуск команд, правки — всё это выполняет обвязка.

Цикл работы агента:

1. Получает промпт.
2. Выбирает инструмент (Read, Edit, Bash, WebFetch).
3. Обвязка выполняет действие.
4. Результат возвращается модели.
5. Модель анализирует результат и решает, что делать дальше.
6. Цикл повторяется, пока задача не выполнена.

Понимать цикл важно по трём причинам. Предсказуемость: вы видите, какие инструменты Claude выберет, и можете направить его — например, понимаете, почему он читает 10 файлов перед правкой одного (ищет контекст). Отладка: если Claude зациклился — значит проблема в неясных требованиях, а не в коде, и вы знаете, на каком шаге сломалось. Контроль: Escape прерывает текущий шаг цикла, не теряя контекст.

**Plan Mode: что это**

Plan Mode — read-only режим: Claude ищет файлы, читает код, задаёт уточняющие вопросы, но не вносит изменения. Ничего не редактирует, не создаёт файлы и не запускает команды оболочки.

Включение:
- `Shift+Tab` — переключает режимы по кругу: Default → Accept Edits → Plan → Default. Внизу терминала появляется индикатор `plan mode on`.
- `claude --permission-mode plan` — запуск сразу в Plan Mode из терминала.
- В VS Code — клик на индикатор режима внизу промпт-бокса (Default / Plan / Auto-Accept). Настройка по умолчанию: `claudeCode.initialPermissionMode` в Settings.

Ревью плана через `Ctrl+G`: после того как Claude составил план, эта горячая клавиша открывает его в текстовом редакторе. План — черновик, Claude ждёт одобрения. Можно добавлять шаги, удалять лишнее, менять порядок, прописывать ограничения. В VS Code план открывается как markdown-документ с возможностью инлайн-комментариев.

**Цикл Explore → Plan → Implement → Commit**

- **Explore** (Plan Mode): Claude читает файлы, прослеживает потоки данных, задаёт вопросы.
- **Plan** (Plan Mode): создаёт план. `Ctrl+G` для ревью. Вы корректируете.
- **Implement** (Normal Mode): `Shift+Tab` → Normal. Claude реализует утверждённый план.
- **Commit** (Normal Mode): `/commit` — Claude анализирует diff и предлагает описательное сообщение коммита.

**Интервью-подход**

В Plan Mode Claude задаёт уточняющие вопросы вместо того, чтобы угадывать: «HTML или plain text?», «Что делать, если API падает?», «Все поля или только активные?». Каждый ответ убирает ошибочное допущение. Шаблон промпта: «Проведи интервью по требованиям перед созданием плана — задавай уточняющие вопросы по сложным частям, не задавай очевидных».

**Когда Plan Mode нужен**

- Незнакомая кодовая база — нужно понять паттерны до изменений.
- Сложный рефакторинг — изменения в 3+ файлах с зависимостями.
- Размытые требования — «добавить оплату» без уточнения модели, возвратов, edge cases.
- Большие архитектурные решения — выбор БД, стороннего сервиса, паттерна.

**Когда Plan Mode не нужен**

- Однострочные правки: опечатки, импорты, константы.
- Ошибка с очевидной причиной: стек-трейс указывает на файл и строку.
- Все решения уже приняты: «замени fetch на axios, сохрани типы».
- Быстрый тест: можете описать diff одним предложением — пропускайте план.

**Ловушки Plan Mode**

1. *Забыть переключиться обратно.* Claude будет уточнять план бесконечно. Когда план готов — `Shift+Tab` → Normal Mode → «Implement the plan».
2. *Устаревание плана.* Если изменили код между планом и реализацией, план может ссылаться на несуществующие файлы. Выполняйте планы сразу.
3. *Перепланирование.* План — 2–5 минут, не 20. Детали Claude решит при реализации.
4. *Расход контекста.* Чтение файлов в Plan Mode заполняет контекстное окно. Для масштабных исследований — субагенты.

**Planning vs Thinking**

- *Planning* (Plan Mode) — про **ширину**: много файлов, зависимости, архитектура. Read-only, создание плана.
- *Thinking* (Extended Thinking) — про **глубину**: сложная логика, отладка, алгоритмы. Дополнительные thinking-токены для рассуждений.

Можно совмещать: Plan Mode + «ultrathink» — глубокий анализ при создании плана.

**Передача контекста через @-упоминания (VS Code)**

Вместо описания файлов словами — ссылайтесь напрямую. Claude прочитает содержимое и сфокусируется на нужном.

| Синтаксис | Что делает |
|---|---|
| `@auth.js` | Файл. Fuzzy matching: `@auth` найдёт `auth.js`. |
| `@src/components/` | Папка. Trailing slash обязателен. |
| `@app.ts#5-10` | Конкретные строки файла. |
| `@terminal:dev-server` | Вывод терминала по имени. |

Пример промпта:
```
Посмотри @src/api/auth.ts#12-45 и @src/middleware/session.ts.
Middleware не обновляет токены при истечении. Исправь.
```

`Option+K` / `Alt+K` — вставить @-mention с путём и строками из выделения в редакторе.

**IDE-интеграции**

*VS Code Extension.* Установка из Marketplace. 4 способа открыть Claude: Spark Icon вверху справа, Activity Bar слева, Command Palette (`Cmd+Shift+P` → «Claude Code»), Status Bar внизу справа.

Возможности: визуальные диффы side-by-side в редакторе, @-mentions, параллельные разговоры (`Ctrl+Shift+Esc` — новая вкладка с отдельным контекстом; индикатор: синий — ждёт разрешения, оранжевый — закончил), Plan Mode как markdown-документ.

Горячие клавиши:

| Клавиша | Действие |
|---|---|
| `Ctrl+Esc` / `Cmd+Esc` | Переключить фокус: редактор ↔ Claude |
| `Ctrl+Shift+Esc` | Новый параллельный разговор (вкладка) |
| `Ctrl+N` / `Cmd+N` | Новый разговор (при фокусе на Claude) |
| `Option+K` / `Alt+K` | @-mention из выделения |

Настройки расширения: `selectedModel` (модель для новых разговоров), `useTerminal` (текстовый интерфейс), `initialPermissionMode` (`default` / `plan` / `acceptEdits` / `bypassPermissions`), `autosave` (автосохранение перед чтением/записью).

*JetBrains Plugin* (IntelliJ, PyCharm, WebStorm, GoLand, Android Studio). Установка через Marketplace, может потребоваться перезапуск IDE дважды. `Cmd+Esc` — открыть Claude. `Cmd+Option+K` / `Alt+Ctrl+K` — вставить ссылку на файл с номерами строк. `/ide` — подключить Claude из внешнего терминала к JetBrains.

Особенности: автоматическая диагностика (lint-ошибки и syntax errors из IDE передаются Claude), нативный diff viewer, выделение кода → автоматическая передача контекста. Фикс Escape: Settings → Tools → Terminal → снять «Move focus to the editor with Escape».

*Один движок, разные интерфейсы.* CLI, VS Code и JetBrains используют один и тот же процесс Claude Code. Общие настройки: CLAUDE.md, settings.json, MCP-серверы, хуки. Общая история: начали в VS Code → продолжили в CLI через `claude --resume`.

**Контрольные точки и откаты**

Перед каждым действием Claude автоматически создаёт чекпоинт — точку отката. Доступ: двойной `Escape` или `/rewind`. Хранение 30 дней (настраивается через `cleanupPeriodDays`).

В CLI `/rewind` откатывает код и разговор вместе или по отдельности. В VS Code: `Fork conversation from here` (новая ветка, код не трогаем), `Rewind code to here` (откатить файлы, разговор сохранить), `Fork conversation and rewind code` (и то и другое).

Ограничения: файлы, изменённые через Bash-команды (`sed`, `npm init`), не отслеживаются — только инструменты Edit/Write. Ручные правки между сообщениями Claude не захватываются. Чекпоинты — локальный undo в рамках сессии, не замена Git. Перед рискованными изменениями: коммит через Git → Plan Mode → реализация.

**Удалённое управление сессией (Remote Control)**

RC — мост между локальной сессией Claude Code и любым устройством. Код выполняется на вашей машине; телефон, планшет или браузер работают как пульт.

Особенности: полная синхронизация терминала, браузера и телефона в реальном времени; автопереподключение после короткого сна ноутбука; таймаут ~10 минут без связи; только outbound HTTPS, входящие порты не открываются.

Три способа запустить:
1. `claude remote-control --name "Проект"` — серверный режим. Терминал ждёт подключений, показывает QR-код и URL.
2. `claude --remote-control` — обычная интерактивная сессия с включённым RC.
3. `/remote-control` (или `/rc`) — включить RC из уже запущенной сессии без прерывания работы.

Подключение: отсканировать QR камерой (откроется Claude app или браузер), скопировать URL в браузер на любом устройстве, или открыть `claude.ai/code` — сессия в списке с зелёным «online». Одно удалённое подключение за раз в интерактивном режиме; серверный режим допускает несколько.

Флаги: `--name "Проект"` (имя в списке), `--spawn same-dir` / `--spawn worktree` (общая папка или отдельный worktree на сессию), `--capacity 5` (максимум параллельных сессий, по умолчанию 32), `--sandbox` (изоляция файловой системы и сети). `/rename "Дайджест"` — переименовать запущенную. `/config` → «Enable Remote Control for all sessions» → `true` — включить для всех сессий навсегда.

Требования: подписка Pro / Max / Team / Enterprise (API-ключи не работают), аутентификация через `/login`, Claude Code v2.1.51+, workspace trust для папки проекта.

RC vs Claude Code on the web: RC — выполнение на вашей машине, доступ к локальным файлам, MCP, Skills, CLAUDE.md. Claude Code on the web — выполнение в облаке Anthropic, без доступа к локальным файлам. Нужна локальная среда — RC; задача без локальных зависимостей — web.

**Чеклист Plan Mode**

Задача затрагивает 3+ файла или требования размыты — Plan Mode. `Shift+Tab` дважды (из Default), `Ctrl+G` для ревью. После утверждения — `Shift+Tab` → Normal Mode → «Implement the plan». Быстрый тест: можете описать diff одним предложением — пропускайте план.

## Занятие 4. Контекст

### Теория

#### Из чего состоит контекстное окно

Контекст — это всё, что Claude «видит» в моменте: системный промпт, описания инструментов, CLAUDE.md, файлы памяти, разговор (промпты, ответы, прочитанные файлы, выводы команд). У Opus и Sonnet окно — 1 000 000 токенов, у Haiku — 200 000. Из миллиона ~15K зарезервировано под буфер ответа, ещё ~15–20K съедают системные инструкции и описания инструментов. Реальный рабочий объём — около 950K, и заполняется он быстрее, чем кажется.

Команда `/context` показывает раскладку: System prompt, System tools, Custom agents, Memory files, Messages, Free space. Это первая команда, которую стоит освоить — без неё вы не понимаете, на что уходит окно.

#### Почему раздутый контекст снижает точность

**Context Rot** — деградация качества при росте контекста. Бюджет внимания модели конечен, и каждый новый токен расходует его. На 20% заполнения Claude точен, на 50% — рабочий, на 70% начинает терять фокус, на 85% пропускает инструкции, на 95% срабатывает автокомпрессия. Качество падает резко, а не плавно.

**Context Bloat** — мусор в окне: раздутый CLAUDE.md, лишние MCP-серверы (каждый добавляет описания инструментов на 3–7K токенов), нечищеная история с неудачными попытками. Результат: Claude равномерно нарушает все правила, а не только новые.

**Правило 70%**: при многофайловой работе сжимайте контекст или начинайте новую сессию на отметке 70%, не дожидаясь автокомпрессии.

Важный нюанс: prompt caching экономит деньги, но не место в окне. Кэшированные токены всё равно занимают контекст.

#### Что засоряет контекст

- 500 строк кода ≈ 3–5K токенов. 20 файлов = 60K+. «Разберись в проекте» — самая дорогая фраза.
- Вывод команд: `npm test` с verbose, `git log` с диффами, сборки — тысячи токенов каждый.
- Неудачные попытки. Три подхода к багу могут стоить дороже итогового решения.
- MCP-серверы. Проверить стоимость: `/mcp`. Отключайте неиспользуемое.

#### Команды управления

- `/compact` — сжать историю в саммари. Технически записывает `compact_boundary`, локальная история не удаляется, но в API уходят только саммари + новые сообщения. Сокращение типично 80–90%.
- `/compact "фокус"` — сжать с приоритетом на тему. Без фокуса Claude сжимает равномерно, что хуже.
- `/clear` — полный сброс. Нужен между несвязанными задачами, после 2+ исправлений одной ошибки (контекст отравлен), когда Claude путает контексты прошлой задачи и текущей.
- `/rewind` (или двойной Esc) — откат к чекпоинту. Пять действий: откатить код+разговор, только разговор, только код, Summarize from here, Never mind. Чекпоинты живут 30 дней. Файлы, изменённые через Bash, не отслеживаются.
- `/model` — переключение Opus/Sonnet/Haiku внутри сессии. Haiku для быстрых вопросов, Sonnet для повседневной работы, Opus для архитектуры и многофайловых изменений.

**Автокомпрессия** срабатывает на 80–85% (раньше было 95%, но вызывало сбои). Сначала чистятся старые выводы инструментов, потом суммируется разговор. Опасность: срабатывает посреди задачи, вы не контролируете, что сохранится, детали из начала сессии могут исчезнуть.

**Compact Instructions** — секция в CLAUDE.md, которая управляет сжатием. CLAUDE.md полностью переживает компрессию: перечитывается с диска. Пример:

```markdown
# Compact Instructions
When compacting, always preserve:
- The full list of modified files
- Any test commands and their results
- Current task objectives
- Key decisions made during session
```

**Четыре паттерна управления сессиями**:
1. **Task Sandwich**: `/clear` → конкретный промпт → работа → `/compact` или `/clear`.
2. **Research → Implement**: сессия 1 — Plan Mode, план в `PLAN.md`; `/clear`; сессия 2 — реализация по плану в чистом контексте.
3. **Checkpoint Compact**: после каждой завершённой фазы — `/compact "фаза X done, сохрани Y"`. Каждый compact как сохранение в игре.
4. **Subagent Shield**: исследование делегируется субагенту; основной контекст растёт только на размер саммари.

#### Memory: `/memory`, auto-memory, MEMORY.md

Два механизма памяти, оба загружаются при старте:
- `/memory` — открывает файлы памяти для просмотра/редактирования. Переключатель вкл/выкл.
- **Auto-memory** — Claude сам замечает паттерны и сохраняет заметки. Триггер из диалога: «Запомни, что мы используем pnpm». Удаление: «Забудь заметку про…».
- `#` в начале сообщения — быстрая заметка в CLAUDE.md (не в auto-memory).

**Иерархия источников контекста** (от высшего к низшему): Managed policy → проектный CLAUDE.md и `.claude/rules/` → пользовательский `~/.claude/CLAUDE.md` → `CLAUDE.local.md` (не в Git) → дочерние CLAUDE.md в подпапках → auto-memory.

**CLAUDE.md vs. auto-memory**:
| | CLAUDE.md | Auto-memory |
|---|---|---|
| Кто пишет | Вы | Claude |
| Содержание | Инструкции и правила | Заметки и открытия |
| Видимость | Вся команда (в Git) | Только вы |
| Загрузка | Полностью при старте | Первые 200 строк MEMORY.md |

**Лимит 200 строк**. Жёсткое правило для MEMORY.md: строка 201 и далее игнорируется при автозагрузке. Тихая потеря: предупреждение легко пропустить. Решение — индексный паттерн.

**MEMORY.md как индекс**:

```markdown
# Project Memory Index

## Build and Test
- Uses pnpm, not npm
- Test runner: Vitest
- See debugging.md for common failures

## Code Patterns
- See patterns.md for repository pattern
- See api-conventions.md for REST structure

## Known Issues
- Stripe webhook drops events if Redis down
```

Тематические файлы (`memory/debugging.md`, `memory/patterns.md`, `memory/api-conventions.md`) не загружаются при старте — Claude читает по необходимости, когда видит ссылку. Так можно хранить неограниченный объём заметок в пределах лимита.

**Организация**: по темам, не хронологически. Плохо: «15 янв узнал, что payment service принимает центы». Хорошо: «Payment: суммы всегда в центах. processPayment(amount) ожидает центы».

**Что сохранять**: команды сборки, решения и причины архитектурных выборов, баги и обходные решения, особенности окружения (порты, зависимости).
**Что не сохранять**: общие советы («пиши чистый код»), секреты и API-ключи, очевидное из кода (структура папок, package.json), to-do и одноразовые инструкции.

**Перенос в CLAUDE.md**: если запись в auto-memory оказалась полезной в 3+ сессиях — это правило, не заметка. «Команда предпочитает named exports» → CLAUDE.md: «Use named exports, never defaults».

#### Продвинутый CLAUDE.md: Progressive Disclosure

Принцип: вместо одного 150-строчного файла — короткий CLAUDE.md (~40 строк) с указателями на детали в `.claude/rules/`. Аналогия: оглавление книги, а не книга целиком.

**Два способа подключения**:
| | `@import` | «See file for details» |
|---|---|---|
| Когда грузится | При старте, всегда в контексте | По решению Claude, при необходимости |
| Расход | Постоянный | Только когда нужно |
| Гарантия применения | Высокая | Средняя |
| Когда использовать | Критичные правила | Справочные, редко используемые |

Формулировка «Before working on [topic], read [file] for current rules» работает лучше, чем просто «See file» — даёт чёткий триггер.

**.claude/rules/** — модульные правила с frontmatter `paths:`. Файлы без `paths:` загружаются при старте, с `paths:` — только при работе с подходящими файлами:

```yaml
---
paths:
  - "src/content/**/*.md"
---
# Стиль статей дайджеста
- Язык: русский. Заголовок до 80 символов.
- Стиль информационный, без маркетинга.
```

Glob-паттерны: `src/content/**/*.md`, `tests/**`, `*.config.*`, `**/*.test.ts`, `**` (всегда).

## Занятие 5. MCP и внешние системы

### Теория

#### Что такое MCP и какую проблему решает

Model Context Protocol — открытый стандарт от Anthropic, выпущенный в ноябре 2024. Это спецификация, единый способ подключать AI-инструменты к внешним сервисам: базам данных, API, файловым системам. Не плагин и не фреймворк — спецификация, которую может реализовать любой. Поддерживают все крупные AI-клиенты: Claude Code, Claude Desktop, Cursor, Continue.

До MCP под каждый сервис писались собственные tools со схемами и функциями: GitHub API, Slack API, Sentry API — всё руками разработчика клиента. При обновлении API приходилось переписывать. MCP выносит работу в отдельный сервер, который пишет поставщик сервиса или сообщество, а не вы.

Tool use — механизм вызова функций моделью — существовал и до MCP. MCP отвечает на другой вопрос: кто пишет инструмент, где он лежит и как его подключить. Под капотом MCP-инструменты — обычные tools, просто живут в отдельном процессе.

Метафора: MCP — USB-C для AI. Один разъём, любое устройство.

#### Архитектура: три роли

- **Host** — приложение-координатор. В нашем случае Claude Code.
- **Client** — коннектор внутри хоста, держит связь с одним сервером. Клиентов столько же, сколько серверов.
- **Server** — программа-обёртка над сервисом, написанная поставщиком или сообществом.

Хост управляет клиентами, клиенты говорят с серверами, серверы делают работу. При старте Claude Code подключается ко всем настроенным серверам, получает от них список инструментов и загружает в контекст. Когда вы пишете промпт, модель решает, что нужен конкретный инструмент, клиент отправляет запрос серверу, сервер возвращает результат.

#### Три примитива: что сервер отдаёт агенту

- **Tools** — функции, которые Claude вызывает сам. Решает модель. Примеры: `tavily-search`, `github_create_issue`, `postgres_query`. На первый вызов Claude спросит подтверждение.
- **Resources** — данные по URI, подставляются в промпт через `@`. Решает пользователь. Синтаксис: `@server:protocol://resource/path`. Набираете `@` — появляется автодополнение со всеми ресурсами. Пример: «Проанализируй `@github:issue://123` и предложи исправление».
- **Prompts** — готовые шаблоны от автора сервера. Вызываются как команды `/mcp__имя-сервера__имя-промпта`. Решает пользователь. Пример: `/mcp__docs__format`.

Один сервер может дать все три примитива.

#### Три транспорта

| Транспорт | Когда использовать | Пример |
|---|---|---|
| **stdio** | Локальные программы на вашей машине | Filesystem, SQLite, локальные скрипты |
| **HTTP** | Удалённые облачные сервисы. Рекомендуется. Поддерживает OAuth | Tavily, GitHub, Notion, Supabase |
| **SSE** | Потоковые подключения с живой подпиской на события | Редкие случаи |

#### Подключение: claude mcp add и .mcp.json

Два основных способа: CLI-команда `claude mcp add` (для разовых подключений) и JSON-конфиг `.mcp.json` или `settings.json` (для команды и сложных случаев). Оба способа пишут в одни и те же конфиги. Третий путь — попросить Claude настроить сервер по документации.

HTTP-сервер:
```
claude mcp add --transport http tavily https://mcp.tavily.com/mcp
```

stdio-сервер:
```
claude mcp add --transport stdio filesystem -- npx -y @modelcontextprotocol/server-filesystem ./src
```

stdio с переменной окружения:
```
claude mcp add --transport stdio --env TAVILY_API_KEY=${TAVILY_API_KEY} tavily -- npx -y tavily-mcp@latest
```

**Порядок флагов критичен.** Все флаги (`--transport`, `--env`, `--scope`, `--header`) идут ДО имени сервера. Двойной дефис `--` отделяет опции Claude от команды запуска. Структура: команда → флаги → имя → `--` → команда сервера. Если флаг стоит после имени — он игнорируется, сервер подключится без ключа.

Пример `.mcp.json` в корне проекта:
```json
{
  "mcpServers": {
    "tavily": {
      "type": "http",
      "url": "https://mcp.tavily.com/mcp"
    },
    "filesystem": {
      "type": "stdio",
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "./src"]
    },
    "database": {
      "type": "stdio",
      "command": "npx",
      "args": ["-y", "@bytebase/dbhub", "--dsn", "${DATABASE_URL}"],
      "env": { "DATABASE_URL": "${DATABASE_URL}" }
    }
  }
}
```

#### Три области (scope)

| Scope | Где хранится | Для кого |
|---|---|---|
| **local** (по умолчанию) | `~/.claude.json`, привязка к пути проекта | Только вы, только этот проект |
| **project** (`--scope project`) | `.mcp.json` в корне проекта, коммитится в Git | Вся команда |
| **user** (`--scope user`) | `~/.claude.json`, глобально | Только вы, все ваши проекты |

Для командных проектов используем `project`, чтобы конфиг лежал в репозитории.

#### Безопасность ключей

- Ключи только через `${VAR}` в переменных окружения. Никогда в коде, никогда в Git.
- Реальные значения — в `.env` (обязательно в `.gitignore`) или в переменных ОС.
- Проверяйте источник сервера: официальный реестр, открытый GitHub, проверенный издатель.
- Минимальные права: только чтение, если это возможно.
- Не подключайте MCP к продакшн-базам — только dev или staging.
- `.mcp.json` коммитим в Git (это конфиг, не секрет), `.env` — нет.

OAuth: большинство облачных серверов требуют авторизации. Claude Code поддерживает OAuth 2.0 из коробки. Процесс: `/mcp` → выбрать сервер → Authenticate → браузер → логин. Токен сохраняется, на следующих запусках работает автоматически.

#### Каталоги и где искать серверы

- Официальная документация Claude Code (code.claude.com).
- Официальный репозиторий: `github.com/modelcontextprotocol/servers`.
- **MCP Registry** — официальный реестр.
- **PulseMCP**, **mcp.so**, **Smithery** — крупные сообщественные каталоги, тысячи серверов.

Серверы делятся на официальные (проверены Anthropic), вендорские (GitHub, Stripe, Notion пишут сами) и сообщественные (разное качество). Непроверенный сервер — только в изолированном окружении и с минимальными правами.

#### Выбор с умом и Tool Search

Каждый подключённый MCP добавляет описания всех своих инструментов в контекст. Три-четыре сервера = 10-20 тысяч токенов на описания, ещё до первого промпта.

**Tool Search** разгружает контекст автоматически. По умолчанию MCP-инструменты отложены: Claude грузит только нужные по запросу через индекс. Управление — переменная `ENABLE_TOOL_SEARCH`:
- не задано — отложенная загрузка по умолчанию;
- `true` — то же, но работает и с сторонним `ANTHROPIC_BASE_URL`;
- `auto` — пороговый режим: грузить сразу, если описания умещаются в 10% контекста (`auto:5` — свой процент);
- `false` — все инструменты грузятся сразу.

Лимиты вывода: предупреждение на 10K токенов, максимум 25K по умолчанию. Настройки сессии:
```
MAX_MCP_OUTPUT_TOKENS=50000 claude
MCP_TIMEOUT=15000 claude
```

#### MCP Inspector

Отладочный инструмент от Anthropic. Веб-интерфейс для тестирования tools, resources и prompts любого сервера:
```
npx @modelcontextprotocol/inspector
# открыть http://localhost:6274
```

#### Команды управления

CLI (вне Claude Code):
- `claude mcp list` — список серверов;
- `claude mcp get имя` — детали;
- `claude mcp remove имя` — удаление;
- `claude mcp add-from-claude-desktop` — импорт из Claude Desktop.

Внутри Claude Code — команда `/mcp`: интерактивный интерфейс с переподключением, авторизацией, просмотром деталей.

#### Когда MCP, когда нет

| Ситуация | Решение |
|---|---|
| Поиск по файлам проекта | Read, Grep, Bash — встроенные |
| Коммит и пуш в Git | Bash + git, MCP не нужен |
| Внешние сервисы | Tavily, GitHub, Supabase, Sentry — MCP |
| Массовая обработка данных | Python-скрипт, не MCP |
| База данных | Supabase/Postgres MCP |

MCP не годится для пайплайнов на 100K строк или 500 API-вызовов — пишите скрипт. Первый запуск stdio-сервера медленный (npx скачивает пакет). Данные от MCP перепроверяйте.

## Занятие 6. Навыки и автоматизация

### Теория

#### Промпт, навык, хук — три уровня надёжности

- **Промпт** — пожелание. «Запускай Prettier после правок» работает, пока Claude помнит. После длинной сессии или `/compact` — забывает.
- **Навык (skill)** — сохранённая процедура. Один и тот же промпт превращается в команду с ограничениями инструментов, аргументами, динамическими данными.
- **Хук** — гарантия на уровне системы. Срабатывает на событие Claude, выполняется всегда, не зависит от памяти модели.

#### Что такое навык и где он живёт

Навык — папка внутри `.claude/skills/`. Имя папки становится слеш-командой: `.claude/skills/deploy/SKILL.md` → `/deploy`. Рядом с `SKILL.md` можно держать вспомогательные файлы (шаблоны, чеклисты), доступные через `${CLAUDE_SKILL_DIR}`.

Три области видимости:
- `~/.claude/skills/` — Personal, только вы, во всех проектах.
- `.claude/skills/` — Project, едет в Git, работает у всей команды.
- Enterprise — управляется администратором.

#### Структура SKILL.md

Файл начинается с YAML-frontmatter между двумя `---`, дальше — текст инструкции для Claude.

**Основные поля frontmatter:**

| Поле | Назначение |
|------|------------|
| `name` | Имя команды. Строчные буквы и дефисы, до 64 символов. |
| `description` | По этому полю Claude решает, грузить ли навык автоматически. Чем конкретнее — тем точнее автоподбор. |
| `argument-hint` | Подсказка автодополнения: `[env]`, `[staging\|production]`. |
| `allowed-tools` | Инструменты, разрешённые без подтверждения. Не ограничение, а выдача разрешений. |
| `disable-model-invocation` | `true` — только ручной вызов. Обязательно для побочных эффектов: деплой, коммит, внешние API. |
| `user-invocable` | `false` — скрыть из меню `/`, Claude грузит сам по описанию. |

**Продвинутые поля:**

| Поле | Назначение |
|------|------------|
| `model` | Переопределяет модель для навыка. |
| `effort` | Уровень усилий: `low`, `medium`, `high`, `max`. |
| `context: fork` | Изоляция: навык выполняется в субагенте с отдельным контекстом. Основная сессия не засоряется. |
| `agent` | Тип субагента при `context: fork`. |
| `hooks` | Хуки на жизненный цикл навыка. |
| `paths` | Glob-паттерны: навык активен только при работе с указанными файлами. |
| `shell` | `bash` (по умолчанию) или `powershell`. |

**Паттерны `allowed-tools`:**
- Только чтение: `Read Grep Glob`.
- Ограниченный Bash: `Bash(npm test *) Bash(npm run lint*) Read`.
- Полное редактирование: `Read Write Edit Bash Glob Grep`.

#### Динамические данные и аргументы

Обратные кавычки с восклицательным знаком запускают команду в терминале до того, как Claude увидит текст навыка. Вывод подставляется вместо плейсхолдера:

```yaml
---
name: status-report
description: Weekly status with live data
---
Current branch: !`git branch --show-current`
Last 5 commits:
!`git log --oneline -5`
```

**Переменные:**
- `$ARGUMENTS` — все аргументы строкой.
- `$ARGUMENTS[0]` или `$0` — первый аргумент, `$1` — второй.
- `${CLAUDE_SESSION_ID}` — ID сессии.
- `${CLAUDE_SKILL_DIR}` — путь к папке навыка для чтения вспомогательных файлов.

#### Запуск, автоподбор, повторы

- **Ручной**: `/deploy staging`. Надёжнее при большом количестве навыков.
- **Автоматический**: Claude читает `description` всех навыков и грузит подходящий, когда задача совпадает с триггером. Расплывчатое описание = промахи.
- **Повтор по расписанию**: `/loop 5m /digest` — каждые 5 минут. `/loop 10m "проверь статус деплоя"` — произвольный промпт. Без интервала — 10 минут по умолчанию. Применения: мониторинг CI, периодический контент, проверка логов.
- **`/skills`** — список доступных навыков (Personal, Project, Bundled).
- **`/agents`** — менеджер агентов и навыков, генерация конфигурации по описанию.

Встроенные навыки: `/batch`, `/debug`, `/simplify`, `/loop`, `/claude-api`.

#### Что такое хук и зачем он нужен

Хук — внешний обработчик, который Claude Code запускает на событие. В отличие от промпта, хук — часть рантайма, не зависит от внимания модели.

**Жизненный цикл — ключевые события:**

| Событие | Может блокировать | Применение |
|---------|------------------|------------|
| `SessionStart` | Нет | Восстановить контекст после `/compact`. |
| `UserPromptSubmit` | — | Пользователь отправил промпт. |
| `PreToolUse` | Да | Запрет опасных команд, защита файлов. |
| `PostToolUse` | Нет | Автоформатирование, логирование. |
| `Stop` | Да | Линтер, тесты, валидация перед завершением. |
| `Notification` | Нет | Уведомления в Telegram/системные. |

Полная палитра событий шире: инструменты (`PostToolUseFailure`, `PermissionRequest`, `PermissionDenied`), сессия (`SessionEnd`, `StopFailure`), субагенты (`SubagentStart`, `TaskCreated`, `TaskCompleted`), файлы (`FileChanged`, `PreCompact`, `PostCompact`), диалоги (`Elicitation`).

#### Четыре типа хуков

| Тип | Что делает | Таймаут |
|-----|-----------|---------|
| `command` | Shell-скрипт. JSON на stdin, exit code на выходе. | 600 сек |
| `http` | POST на URL. Аудит, облачные функции. | 600 сек |
| `prompt` | Одношаговая LLM-проверка (`ok: true/false`). Haiku по умолчанию. | 50 сек |
| `agent` | Субагент с инструментами (Read, Grep, Glob). До 50 шагов. | 60 сек |

#### Конфигурация и протокол общения

Хуки описаны в `settings.json`. Три файла:
- `~/.claude/settings.json` — личные, для всех проектов, не в Git.
- `.claude/settings.json` — командные, попадают в Git.
- `.claude/settings.local.json` — локальные, не в Git.

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          { "type": "command", "command": "/abs/path/block-dangerous.sh" }
        ]
      }
    ]
  }
}
```

**Матчеры:** `"Bash"`, `"Edit|Write"` (pipe = OR), `"mcp__.*"` (все MCP), `"compact"`. Без матчера — хук срабатывает на все события этого типа.

**Протокол command-хука:**
- На stdin приходит JSON: `tool_name`, `tool_input`, `stop_hook_active`.
- Exit 0 — разрешить, exit 2 — заблокировать, любой другой — предупреждение.
- В stdout можно вернуть JSON: `{"permissionDecision": "deny", "reason": "..."}` или `additionalContext` — добавить текст в контекст Claude.

#### Применения хуков

- **Автоформатирование** (PostToolUse, `Edit|Write`): извлечь путь через `jq`, прогнать через Prettier. PostToolUse не отменяет действие — файл уже записан.
- **Блокировка опасных команд** (PreToolUse, `Bash`): grep по списку паттернов (`rm -rf`, `sudo`, `git push --force`), exit 2 при совпадении.
- **Защита файлов** (PreToolUse, `Edit|Write|Read`): блокировка `.env`, `.git/`, lock-файлов.
- **Валидация перед завершением** (Stop): проверка тестов, наличия обязательных полей, линтера.

**Подводные камни:**
- Stop-хук без guard `stop_hook_active` → бесконечный цикл (Claude завершает → блок → исправляет → блок). Guard разрывает цикл.
- Скрипты должны быть исполняемыми (`chmod +x`), пути — абсолютные.
- `jq` обязателен для парсинга stdin. `/hooks` показывает настроенные хуки, редактирование — только вручную в settings.json.

#### TDD-цикл с Claude

Без тестов «готово» определяется на глаз. С тестами Claude получает чёткую цель: запустил, увидел ошибку, исправил, проверил. Тесты не дают добавлять лишнее.

**RED → GREEN → REFACTOR:**

1. **RED.** «Напиши ПАДАЮЩИЙ тест для X. НЕ пиши реализацию. Тест должен падать.» Без явного запрета Claude напишет тест и код одновременно.
2. **GREEN.** «Напиши минимальный код, чтобы тест прошёл. Только код для текущего теста, ничего лишнего.» Без слова «минимальный» появятся валидация, обработка ошибок, лишние методы.
3. **REFACTOR.** «Улучши код, не ломая тесты. Фокус: читаемость / дублирование / производительность.» Если тесты сломались — откат.

**Вертикальная нарезка** (рекомендуется): один тест → одна реализация → повтор. **Горизонтальная** (плохо): все тесты сразу — Claude подсматривает и подгоняет.

**Context pollution.** Все фазы в одном контексте — анализ автора тестов просачивается в реализацию. Решение: `context: fork` в TDD-навыке или ручное разделение через `/clear`.

**Антипаттерны:** Claude правит тест вместо кода (явно: «Исправь реализацию, НЕ тест»), дублирует код в тестах, моки отрываются от реальности (импортируйте типы), `/compact` теряет контекст тестов.

**Цель:** настроить два хука — автоформатирование Prettier на `PostToolUse` и блокировку опасных Bash-команд на `PreToolUse`.

**Задание:**

2. Создать `.claude/hooks/block-dangerous.sh`, сделать исполняемым (`chmod +x`).

4. Проверить блокировку: попросить Claude `rm -rf node_modules` или `git push --force` — должно сработать «Blocked: ...».
5. Проверить безопасные команды (`ls`, `git status`) — выполняются.
6. Проверить автоформатирование: создать JS-файл — Prettier причешет его автоматически.
7. Открыть `/hooks` — оба хука с источником `[Project]`.

**Скрипт `block-dangerous.sh`:**
```bash
#!/bin/bash
CMD=$(jq -r '.tool_input.command' 2>/dev/null)
[ -z "$CMD" ] || [ "$CMD" = "null" ] && exit 0
for PATTERN in "rm -rf" "sudo" "chmod 777" \
  "drop table" "git push --force" "git reset --hard"; do
  if echo "$CMD" | grep -qi "$PATTERN"; then
    echo "Blocked: $PATTERN" >&2
    exit 2
  fi
done
exit 0
```

**`.claude/settings.json`:**
```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          { "type": "command",
            "command": "jq -r '.tool_input.file_path' | xargs npx prettier --write 2>/dev/null; exit 0" }
        ]
      }
    ],
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          { "type": "command",
            "command": "/АБС/ПУТЬ/.claude/hooks/block-dangerous.sh" }
        ]
      }
    ]
  }
}
```

**Критерий выполнения:** опасные команды блокируются с понятным сообщением, безопасные проходят, файлы автоматически форматируются после записи, `/hooks` показывает оба хука.

   - REFACTOR: «Улучши код, не ломая тест. Фокус: читаемость.»
3. **Цикл 2 — описание:**
   - RED: новый падающий тест на `result.description` (первый параграф после h1). Реализацию не трогать.
   - GREEN: «Напиши минимальный код, чтобы оба теста прошли. Исправь реализацию, НЕ тест.»
   - REFACTOR: убрать дублирование.
4. Финал: `npx vitest run --reporter=verbose` — оба теста зелёные.

**Точные промпты:**

| Фаза | Промпт |
|------|--------|
| RED | «Напиши ПАДАЮЩИЙ тест для [X]. НЕ пиши реализацию. Тест должен падать.» |
| GREEN | «Напиши минимальный код, чтобы тест прошёл. Только код для текущего теста, ничего лишнего.» |
| REFACTOR | «Улучши код, не ломая тесты. Фокус: [читаемость / дублирование / производительность]. Запусти тесты после изменений.» |

**Правила:** один тест за раз (vertical slicing), фраза «Исправь реализацию, НЕ тест» обязательна, ключевое слово «минимальный» обязательно, тест — спецификация, не подстраивается под код.

**Критерий выполнения:** два теста проходят (`2 passed`), реализация содержит ровно столько кода, сколько нужно для тестов, ни один цикл не объединял написание теста и реализации в одном промпте.

### Упражнение 6.4: Навыки `/digest` и `/cover` + Stop-хук валидации

**Цель:** собрать продакшн-цепочку для дайджеста: один навык вызывает другой, Stop-хук гарантирует полноту статьи.

**Задание:**
1. Подключены MCP Tavily и Replicate (`/mcp` показывает `connected`), редполитика — в `CLAUDE.md`.
2. Создать `/cover` — генерация обложки через Replicate (`black-forest-labs/flux-schnell`, 16:9), сохранение в директорию статьи.
3. Создать `/digest` — полный цикл: поиск через Tavily → выбор новости → статья по редполитике → вызов `/cover` → frontmatter с обложкой → коммит и пуш → отчёт.
4. Создать Stop-хук `validate-article.sh` с проверкой `title`, `description`, `cover`/`image`, `source`/URL и guard `stop_hook_active`.
5. Прописать Stop-хук в `.claude/settings.json` (без матчера), путь — абсолютный.
6. Запустить `/digest`, выпустить 2-3 дайджеста.

**`.claude/skills/cover/SKILL.md`:**
```yaml
---
name: cover
description: Generate cover image for a digest article via Replicate
disable-model-invocation: true
argument-hint: [article description]
---

Generate a cover image for: $ARGUMENTS
1. Compose an English image prompt. Style: modern, minimalist, tech-oriented illustration. No text on image.
2. Use the Replicate MCP server with model `black-forest-labs/flux-schnell`,
   num_outputs=1, aspect_ratio="16:9".
3. Save the image to the article's directory as cover.webp.
4. Report the file path.
```

**`.claude/skills/digest/SKILL.md`** — пошаговая инструкция: Tavily search → write → invoke `/cover <one-sentence description>` → add cover to frontmatter → commit `feat: add digest — <title>` → push → report.

**Stop-хук (ключевой фрагмент):**
```bash
#!/bin/bash
ACTIVE=$(jq -r '.stop_hook_active' 2>/dev/null)
[ "$ACTIVE" = "true" ] && exit 0   # guard от бесконечного цикла

ARTICLE=$(find content/ -name "*.md" -newer .git/index 2>/dev/null | head -1)
[ -z "$ARTICLE" ] && exit 0

ERRORS=""
grep -q "^title:" "$ARTICLE" || ERRORS="${ERRORS}Missing title. "
grep -q "^description:" "$ARTICLE" || ERRORS="${ERRORS}Missing description. "
grep -qE "^(cover|image):" "$ARTICLE" || ERRORS="${ERRORS}Missing cover. "
grep -qE "(^source:|https?://)" "$ARTICLE" || ERRORS="${ERRORS}Missing source. "

if [ -n "$ERRORS" ]; then
  echo "Article validation failed: ${ERRORS}" >&2
  exit 2
fi
exit 0
```

**Конфиг Stop-хука в `.claude/settings.json`:**
```json
"Stop": [
  {
    "hooks": [
      { "type": "command",
        "command": "/АБС/ПУТЬ/.claude/hooks/validate-article.sh" }
    ]
  }
]
```

**Критерий выполнения:** `/digest` запускает полный цикл одной командой; внутри корректно вызывается `/cover`; Stop-хук блокирует завершение, пока в статье нет заголовка/описания/обложки/ссылки на источник; guard `stop_hook_active` не даёт зациклиться; `/hooks` показывает Stop-хук с источником `[Project]`; статья опубликована на Vercel со всеми обязательными элементами.

## Занятие 7. Агенты и оркестрация

### Теория

#### Зачем нужны субагенты

Основная сессия Claude — это один контекст, в который оседает всё: прочитанные файлы, вывод тестов, логи, диффы. На 70% заполнения качество резко падает — Claude «забывает» ранние решения и хуже следует инструкциям. Если для одной задачи нужно прочитать 50 файлов, контекст забивается мусором, мешающим следующей задаче.

Решение — выносить разовые подзадачи в отдельные сессии. Субагент — это отдельный процесс Claude со своим контекстом и своим набором инструментов. Получает задачу от основной сессии, выполняет, возвращает только результат. После завершения закрывается, его контекст не попадает в основную сессию.

Три преимущества:
- **Чистый основной контекст** — субагент, прочитавший 50 файлов, возвращает 6 абзацев саммари.
- **Безопасность через инструменты** — read-only ревьюер физически не может править код.
- **Экономия на модели** — Haiku для поиска в 5–10 раз дешевле Opus.

#### Встроенные субагенты

| Тип | Модель | Инструменты | Назначение |
|---|---|---|---|
| Explore | Haiku | Только чтение | Поиск и исследование в кодовой базе |
| Plan | Наследует | Только чтение | Анализ для Plan Mode без правок |
| General-purpose | Наследует | Все | Сложные многошаговые операции |

Explore имеет три уровня глубины: `quick` (поиск по ключам), `medium` (несколько файлов), `very thorough` (полный анализ). Начинать с quick — выше уровень, больше токенов и времени.

#### Свои агенты в `.claude/agents/`

Кастомный агент — markdown-файл с frontmatter и системным промптом. Области хранения по приоритету:

| Область | Путь | Охват |
|---|---|---|
| CLI-флаг `--agents '{...}'` | — | Только текущая сессия |
| Управляемые корпоративные | — | Высший приоритет, политики организации |
| Проектные | `.claude/agents/` | Все, кто клонирует репо |
| Личные | `~/.claude/agents/` | Все ваши проекты |
| Плагины | в папке плагина | Где плагин включён |

**Базовые поля frontmatter:** `name` (lowercase + дефисы), `description` (критично для автоделегирования), `tools` (allowlist разрешённых), `disallowedTools` (denylist), `model` (sonnet/opus/haiku/inherit), `permissionMode` (default/acceptEdits/plan).

**Продвинутые:** `maxTurns` (потолок шагов), `background: true` (всегда в фоне), `memory` (user/project/local — персистентная память), `isolation: worktree` (отдельная копия репо), `mcpServers`, `hooks`.

Без `tools` агент наследует всё. Allowlist безопаснее, denylist удобнее когда нужно «всё, кроме двух».

Пример:
```yaml
---
name: codebase-researcher
description: Исследование кодовой базы перед рефакторингом. Использовать проактивно.
tools: Read, Grep, Glob, Bash
model: haiku
permissionMode: plan
maxTurns: 40
---
```

#### Память агента

Три режима хранения: `user` (`~/.claude/agent-memory/<name>/`), `project` (`.claude/agent-memory/<name>/`, коммитится), `local` (`.claude/agent-memory-local/<name>/`, в `.gitignore`).

#### Автоделегирование vs ручной вызов

Автоматически: Claude читает description и решает, запускать ли. Качество описания критично.
Вручную: «Use the code-reviewer subagent to review my recent changes». Надёжнее при большом числе агентов — иначе Claude путается. Можно параллельно: «Run code-reviewer and test-runner in parallel».

#### Фоновые задачи и task list

- `/agents` — интерактивный менеджер.
- `claude agents` — список из CLI.
- `Ctrl+T` — показать/скрыть task list со статусами фоновых задач.
- `Ctrl+B` — отправить текущую задачу в фон.

**Правило 30 секунд:** задача дольше — в фон. Фоновые субагенты не задают вопросов, весь контекст в промпт заранее. Один уровень вложенности — субагент не может создать субагента.

#### Agent Teams

Ограничение субагентов: А не видит, что делает Б. Обмен — только через основную сессию. Для совместной работы нужны Agent Teams — несколько полноценных сессий Claude Code одновременно с общим task list, зависимостями и прямой перепиской.

| Критерий | Субагенты | Agent Teams |
|---|---|---|
| Коммуникация | Только с основной | Между собой напрямую |
| Координация | Основная сессия | Самокоординация через task list |
| Жизненный цикл | Задача → ответ → завершение | Длительная работа |
| Стоимость | Дешевле | ~4× дороже соло |
| Когда | Фокусные подзадачи | Параллельная работа с обсуждением |

**Архитектура:** лид (основная сессия) + участники (отдельные сессии Claude Code, каждый со своим контекстом). Общий список задач с зависимостями + почта (прямые сообщения).

**Жизненный цикл задачи:** `pending → in_progress → completed`. Задача с зависимостью не берётся, пока не закрыты предшественники. Закончив — участник автоматически берёт следующую свободную.

**Включение:** `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1` в `settings.json`, требует Claude Code 2.1.32+. Статус — experimental.

**Управление:** `Shift+Down` — переключение между участниками; `Enter` — открыть полную сессию; `Escape` — прервать ход; `Ctrl+T` — task list.

**Размер команды:** 3–5 участников, 5–6 задач на каждого. Меньше — координация дороже выгоды; больше — участник работает долго без контрольных точек.

**Стоимость:** соло ~200K токенов, 3 субагента ~440K (закрываются и освобождают память), team из 3 ~800K (полные устойчивые сессии). Контролировать через `/cost`.

**Паттерны:**
- **Параллельное ревью** — security + performance + tests читают один PR с разных ракурсов, лид сводит.
- **Состязательная отладка** — каждый участник проверяет гипотезу и опровергает соседей через mailbox.
- **Сквозная фича** — backend сообщает API-схему, frontend и tests разблокируются. Правило: каждый правит только свои файлы.
- **Утверждение плана** — участник пишет план, лид утверждает, потом исполнение.

**Грабли:** лид начинает делать сам («Wait for your teammates»); статус «залипает» в `in_progress`; конфликты файлов между участниками — делить работу по папкам.

#### Worktrees: изоляция и параллельный рефакторинг

Проблема: одна ветка — одна задача. Прилетел хотфикс — переключение ломает контекст Claude.

**Git worktree** — отдельная рабочая папка, связанная с тем же репозиторием: своя ветка и staging, общий `.git` и история. Одну ветку нельзя открыть в двух worktree.

`claude --worktree <name>` создаёт `.claude/worktrees/<name>/`, ветку `worktree-<name>` от `origin/HEAD` и стартует сессию. На выходе — диалог keep/remove.

Команды:
- `claude --worktree feature-auth` — именованный worktree.
- `claude -w feature-auth` — короткая форма.
- `claude --worktree feature-auth -p "задача"` — worktree + headless.
- `git worktree add ../my-feature -b feature-branch`, `git worktree list`, `git worktree remove`, `git worktree prune`.

**Гигиена:** добавить `.claude/worktrees/` в `.gitignore`; зависимости устанавливать в каждом отдельно (`npm install`); `.env` и другие untracked-файлы — через `.worktreeinclude` в корне (формат как `.gitignore`, копирует только untracked).

**Subagent + `isolation: worktree`** — каждый запуск агента в своей копии репозитория. Три параллельных мигратора правят три пакета без конфликтов. Если результат не устроил — папка удаляется, остальные сохраняются.

**Пять фаз масштабного рефакторинга:**
1. **План** — Plan Mode анализирует объём, выдаёт план, сохраняем.
2. **Фундамент** — только новый код, ничего не ломающий: типы, утилиты, структуры.
3. **Масштабирование** — обновляем зависимые файлы партиями 3–5, коммит после каждой фазы.
4. **Проверка** — полная сборка, тесты, `tsc --noEmit`.
5. **Зачистка** — мёртвый код, неиспользуемые импорты, TODO.

Правила: типы и логика в разных коммитах; правило 70% (`/compact` или новая сессия); «меняй только то, что указано»; контрольная точка в CLAUDE.md (Фаза / Последнее завершённое / Следующее / Изменённые файлы).

#### Когда какой уровень

- Одна задача в одном предложении → один Claude.
- Список независимых подзадач → субагенты.
- Подзадачам нужно обсуждать и координировать → Agent Teams.
- Изоляция по файлам / параллельные ветки → worktrees.

## Занятие 8. Agent Harness

### Теория

**Что такое harness и зачем он нужен**

Harness (обвязка) — всё, что окружает модель: контекст, инструменты, права, механизмы восстановления. Сама по себе модель без обвязки — собеседник в чате. С обвязкой — рабочий агент, который читает файлы, правит код, ходит во внешние сервисы и возвращается к задаче после сбоя.

Три составляющие:
- **Контекст** — что Claude видит при запуске: CLAUDE.md, файлы, история, Memory.
- **Инструменты** — что Claude может делать: Read, Edit, Bash, MCP-серверы.
- **Восстановление** — как агент ведёт себя после ошибки, выхода за лимит, /compact.

Harness — это всё, что собрано на ступенях 1–7: CLAUDE.md и Permission Modes (1–4), MCP и Tavily (5), Skills и Hooks (6), Subagents и Worktrees (7). Восьмая ступень добавляет к конвейеру отладку, headless-запуск и SDK.

**Принцип минимизации инструментов**

Vercel убрал 80% инструментов из своего агента, оставил bash и чтение файлов. Точность поднялась с 80% до 100% на том же наборе задач, скорость выросла в 3,5 раза. Правило: чем мощнее модель, тем проще должна быть её обвязка. Лишний инструмент — лишний выбор для модели, лишняя ошибка.

**Три способа верификации**

| Вид | Инструменты | Когда |
|---|---|---|
| Rules-based | Линтер, `tsc --noEmit`, тесты | Есть проверяемый способ. Быстро и дёшево. |
| Visual | Скриншоты Playwright MCP | Визуальный контроль UI или вёрстки. |
| LLM Judge | Вторая модель оценивает результат | Нет формальных правил. Медленно, не всегда воспроизводимо. |

Приоритет: rules-based > visual > LLM-judge. Линтеры и тесты — первая линия автопроверки, они дают детерминированный сигнал PASS/FAIL.

**Файловая система как внешняя память**

Длинные задачи не держим в контексте — храним на диске: `progress.md`, `tasks.md`, `decisions.md`. Claude пишет по ходу, читает при необходимости. Контекст остаётся чистым, задачи переживают `/compact` и `/clear`.

**Отладка: цикл и принципы**

Claude ошибается не из-за нехватки интеллекта, а из-за нехватки прозрачности. Цикл из четырёх шагов:
1. **Воспроизвести.** Запустить падающую команду, собрать точный вывод.
2. **Изолировать.** Отдать полный стек-трейс.
3. **Исправить.** Claude предлагает правку, вы проверяете перед применением.
4. **Проверить.** Запустить снова. Новая ошибка → шаг 1 с обновлённым контекстом.

Правило полного контекста: копируйте полный текст ошибки, не пересказывайте. Команда запуска, что изменилось, что в `.env`, релевантные коммиты. Скриншоты — нет, текст — да (его можно искать и парсить).

**Logging Workflow**

Сначала логи — потом фикс. Просим Claude добавить `console.error` в catch-блоки, запускаем, отдаём логи обратно. 30 секунд логов вместо 20 минут гадания.

Поток ошибок одной строкой:
```bash
npm run build 2>&1 | claude -p "Исправь ошибки сборки"
npm test 2>&1 | claude -p "Исправь падающие тесты"
npm run dev 2>&1 | tail -100 | claude -p "Что ломается?"
```

`2>&1` объединяет stderr и stdout — иначе ошибки потеряются.

Шорткат `ask` в `~/.zshrc`:
```bash
ask() { "$@" 2>&1 | claude -p "Проанализируй вывод и предложи исправления."; }
```

**Невидимые ошибки и ложный адрес**

Невидимые: устаревшее состояние в React, политики доступа Supabase. Лечатся только журналом — фиксируем фактическое состояние в момент ошибки. Ложный адрес: падает в пятой функции, сломана первая. Сложные баги — через Plan Mode (Shift+Tab дважды) или Explore-субагента.

**Цикл бесконечных правок**

Claude чинит A → ломает B → чинит B → снова ломает A. Причина не в коде, а в неясных требованиях. Esc, уточняем правильное поведение одной фразой: «Тест ожидает X. Реализация делает Y. Правь тест.» Не помогло за 2–3 попытки — `/clear` и новый промпт со всем, что узнали.

Диагностика: `/doctor`, `claude --verbose`, `/cost`, `/rewind` (или Esc-Esc).

**Headless-режим**

Три способа встроить Claude в процесс:
- `claude -p` — для скриптов, CI/CD, cron.
- GitHub Action `anthropics/claude-code-action@v1` — реакция на `@claude` в PR/issue.
- Hooks — автотриггеры внутри интерактивной сессии.

Ключевые флаги headless:
- `--output-format text|json|stream-json` (для скриптов — `json`).
- `--allowedTools "Read,Edit,Bash(npm test)"` — предодобренные инструменты, в headless нет человека для Allow.
- `--bare` — пропускает autodiscovery (хуки, навыки, MCP, CLAUDE.md). Стабильность в CI.
- `--continue` / `--resume "$id"` — продолжение или возобновление по session_id.
- `--max-turns N` — потолок шагов.
- `--model claude-sonnet-4-6` — выбор модели.

JSON-вывод парсим через jq:
```bash
claude -p "..." --output-format json --bare | jq -r '.result'
```

Slash-команды и навыки в `-p` не работают — описывайте задачу словами.

**GitHub Actions**

Установка: `/install-github-app` в интерактивном Claude Code. Команда поднимает OAuth, кладёт `ANTHROPIC_API_KEY` в secrets, открывает PR с двумя workflow: `claude.yml` (реакция на `@claude`) и `claude-code-review.yml` (автоматический разбор PR на opened/synchronize).

Кастомизация через `claude_args` (в v1 GA старые поля `allowed_tools`, `model` заменены):
```yaml
- uses: anthropics/claude-code-action@v1
  with:
    prompt: "Review PR for security, performance, error handling."
    anthropic_api_key: ${{ secrets.ANTHROPIC_API_KEY }}
    claude_args: |
      --max-turns 5
      --model claude-sonnet-4-6
```

Боевые паттерны: Security Gate (PASS/FAIL, блокирует деплой), аудит `process.env` против `.env.example`, разбор логов Vercel через пайп.

Контроль расходов: `--max-turns` всегда, `timeout-minutes: 10`, deny-список `Read(.env*)`, `Read(**/*secret*)` в `settings.json`, ключи только в secrets.

**Claude Agent SDK**

Библиотека для встраивания Claude Code в свой код. Те же инструменты, тот же цикл, что в интерактиве, но с программным контролем сверх того, что даёт `claude -p`.

Переименование: было `claude-code-sdk`, стало `claude-agent-sdk` (`@anthropic-ai/claude-agent-sdk`).

Два интерфейса:
- `query()` — одноразовая сессия. Контекст сбрасывается после ответа.
- `ClaudeSDKClient` — постоянная сессия, контекст между `query()`, поддерживает `interrupt()`, смену модели и MCP на лету.

Критический нюанс: по умолчанию SDK не читает CLAUDE.md и settings. Чтобы подхватить — `setting_sources=["project"]` плюс `system_prompt={"type": "preset", "preset": "claude_code"}`.

Свои MCP-инструменты через `@tool` и `create_sdk_mcp_server`. Имя инструмента в allowlist: `mcp__<server>__<tool>`.

Когда какой уровень:
- Одно действие — один Claude.
- Список независимых подзадач — субагенты.
- Параллельная изоляция — Agent Teams или worktrees.
- Регулярный запуск — `claude -p` + cron, n8n или `/loop`.

### Упражнение 8.4: Полный автопилот дайджеста через /loop

**Цель:** запустить финальный harness — дайджест выпускается без человека по расписанию.

**Задание:** проверить ручной `/digest` (Agent Teams из четырёх агентов: news-scout, writer, cover-artist, page-builder), убедиться что Stop-хук валидирует поля и PostToolUse-хук пишет лог. Запустить `/loop 30m /digest` и оставить на три часа. Мониторить `/cost`, `/context`, блог и логи. Защититься от перерасхода и пушей в main.

**Конфигурация `.claude/settings.json`:**
```json
{
  "hooks": {
    "PreToolUse": [{ "matcher": "Bash", "hooks": [{ "type": "command",
      "command": "echo \"$CLAUDE_TOOL_INPUT\" | grep -q 'git push.*main' && exit 1 || exit 0" }] }],
    "PostToolUse": [{ "matcher": ".*", "hooks": [{ "type": "command",
      "command": "bash .claude/hooks/pipeline-log.sh" }] }],
    "Stop": [{ "matcher": ".*", "hooks": [{ "type": "command",
      "command": "node .claude/hooks/validate-article.js" }] }]
  }
}
```

Stop-хук проверяет поля `title`, `description`, `cover`, `source`. Exit 2 блокирует завершение, Claude дописывает недостающее. Защита от рекурсии — флаг `stop_hook_active`.

Запуск: `/loop 30m /digest`. Мониторинг: `/cost` каждые 10 минут (≤ $0.30 за выпуск), `/context` (на 70% — `/compact`), `tail -100 logs/pipeline.log`.

Финальная архитектура harness:
```
/loop 30m /digest
  └── /digest = Agent Teams из 4 агентов
  └── Хуки: Stop (валидация) + PostToolUse (лог) + PreToolUse (защита main)
  └── Контекст: CLAUDE.md (редполитика) + Memory (факты)
  └── Инструменты: Tavily + Replicate + GitHub через MCP
  └── Деплой: Vercel на push
```

**Критерий выполнения:** ручной `/digest` доходит до деплоя; Stop-хук блокирует пустое описание; PostToolUse пишет каждый шаг в `logs/pipeline.log`; за 3 часа — 6 выпусков, суммарно ≤ $2; deny-список блокирует чтение `.env`, PreToolUse-хук блокирует прямой push в main.
